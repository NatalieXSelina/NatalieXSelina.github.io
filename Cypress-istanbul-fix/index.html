Cypress-istanbul-fix

Author: NatalieXSelina Date: April 20, 2023  14:58:10

背景

本地调试 排查过程
可以观察到问题的报告：http://frps.sre.jdydevelop.com:7001/job/fx-uitest/job/fx-ui-testing1/Report/src/components/form/formula/index.html

fx-ui-test工程
为了方便调试和观察结果，fx-ui-test工程调整如下：

"scripts": {
    ...
    "cy:run:singlefile": "cypress run --spec cypress/e2e/ui/formula/formula.textarea.spec.js",
    "test:clear":"rm -r allure-results/ allure-report .nyc_output coverage || true",
    "instrument:test":"npm run test:clear && npm run cy:run:singlefile -- --env coverage=true"
},

这样只需要 npm run instrument:test一行命令就可以观察到结果

前端工程的插桩debug
首先webpack配置需要改一下，不然debug比较麻烦

devtool: 'source-map', // 要换成这个
devtool: 'eval-cheap-module-source-map',

这两种配置的区别指路：https://blog.csdn.net/Magneto7/article/details/115339508

改好配置以后打包编译，dist目录下面能看到包含插桩后携带探针的源代码

看了一下fx-code-pc项目webpack打包的规则定义，IP()公式最后打包编译在form.js中，可以看出来函数中是有插入探针的

IP()公式的源代码很简单，大家也可以很明确的看出来探针的插入是正确的

此时本地运行fx-ui-test工程，可以看到覆盖率报告结果显示是正常的（和上述报告链接标注的地方不同）

上述是使用本地开发环境的build\webpack.config.dev.js配置进行打包编译

这时候换成dev环境的webpack文件进行打包编译——build\webpack.config.test.js

会发现覆盖率出现问题

PS：上面是用fx-code-pc排查问题的，但是我发现打包太久了SOS。因此后文换成fx-code-mobile项目调试，打包快一些..

本地开发和dev环境的webpack配置区别
build\webpack.config.test.js加一下devtool: ‘source-map’

比较 fx-code-mobile\src\components\form\formula\Formula.ts 打包后携带探针的代码

本地开发环境：dev.txt

dev环境：test.txt

现象：

两个环境插桩后，覆盖率统计处对于同一份代码的行号不同
本地开发环境有inputSourceMap部分
猜测：是和sourceMap相关的问题

调查：

1、这个inputSourceMap是怎么来的？

node_modules\babel-plugin-istanbul\lib\index.js

在此处加一下调试代码，打印this.file.inputMap

发现使用build\webpack.config.test.js时，this.file.inputMap为null

            if (!inputSourceMap && this.file.inputMap) { // this.file.inputMap为null，这边if判断就是false，就不会走到下一行去
            inputSourceMap = this.file.inputMap.sourcemap;
            }

2、但是为什么会是null呢，明明babel-plugin-istanbul插件已经配置开启sourceMap，webpack打包也加了devtool

那么再看看 webpack.config.test.js 和 webpack.config.dev.js 有什么区别

首先，这两个webpack打包配置都是从common扩展来的，仔细看一下主要是module这边有区别

直观一点，索性直接打印结果来看

对比一下，可以看出来，本地开发是替换为esbuild-loader（并且使用缓存加速）而没有使用ts-loader。查了一下说是esbuild-loader速度更快。

3、那么ts-loader为什么会导致没有取到inputMap呢？而且ts-loader会影响插桩插件的sourceMap获取并没有见到官方说明。

第一个问题查了一下，这个插件是取tsconfig.json文件下的配置，而sourceMap未配置的话默认是false

https://github.com/TypeStrong/ts-loader

调整完以后，再build一次，上文提到的this.file.inputMap能够正常打印了

运行fx-ui-test工程，这回测试报告的结果也是正确~

问题到这里，实际已经解决了源代码的获取问题

4、我觉得还没有完全破案，ts-loader为什么会影响插桩插件的sourceMap获取？

看下ts-loader的处理

node_modules\ts-loader\dist\index.js

这边打印出来的东西就是插桩里面的inputSourceMap信息

那么 babel-plugin-istanbul 中的 inputMap 是由 ts-loader ——> babel-loader ——> babel-plugin-istanbul 这样的路径获取，并通过node_modules\babel-plugin-istanbul\lib\index.js中的方法转化为插桩中的 inputSourceMap 信息
